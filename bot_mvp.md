# AgentLayer Telegram Bot MVP

## Что это за MVP

Это не настоящий wallet и не торговый продукт.

Это Telegram-бот для жюри и demo-сценария, который показывает главную идею AgentLayer:

- пользователь пишет финансовое намерение обычным языком;
- агент понимает запрос;
- агент готовит `preview`;
- пользователь подтверждает действие;
- бот применяет виртуальную операцию;
- пользователь видит обновленный демо-портфель и историю.

Главная цель: за 5 минут показать UX агентного финансового интерфейса.

## Жесткие ограничения MVP

В этой версии должно быть только:

- Telegram
- OpenRouter для общения с моделью
- один backend
- одна локальная база состояния
- фиксированные демо-цены
- фиксированная комиссия

В этой версии не должно быть:

- реальных денег
- реальных кошельков
- onchain
- operator API
- market data API
- webhook orchestration
- Redis
- очередей
- микросервисов
- сложной аналитики

## Главное архитектурное решение

Почти вся conversational-логика может жить в `system prompt`, но состояние нельзя хранить в модели.

Если буквально все будет только в prompt, demo сломается на:

- подтверждении операции
- повторных сообщениях
- пересчете баланса
- истории
- рестарте процесса

Поэтому самый простой и надежный MVP такой:

- OpenRouter отвечает за понимание намерения, уточнение запроса и текст preview
- сам бот локально хранит демо-баланс, pending action и историю операций
- математика по сделке считается локально по фиксированным правилам

Это все еще очень простой MVP, но он не развалится во время демо.

## Рекомендуемый стек

Для этого MVP я рекомендую:

- `Python 3.11`
- `aiogram` для Telegram
- `httpx` для OpenRouter
- `SQLite` для локального состояния
- `Railway worker` или локальный запуск на ноутбуке

Почему не webhook:

- polling проще поднять
- не нужен публичный callback URL
- меньше точек отказа
- для demo это надежнее и быстрее

Если потом понадобится production-режим, можно перейти на webhook. Для жюри это не нужно.

## Минимальная архитектура

```text
Telegram User
    ->
Telegram Bot (aiogram, long polling)
    ->
OpenRouter Chat Completion
    ->
Bot Runtime
    ->
SQLite
```

Что делает каждый слой:

- Telegram Bot: принимает сообщения, показывает кнопки, отдает ответы
- OpenRouter: понимает intent и возвращает структурированный ответ
- Bot Runtime: проверяет правила, строит preview, применяет confirm, обновляет баланс
- SQLite: хранит пользователя, портфель, pending action и историю

## Правильная граница ответственности

### OpenRouter делает

- распознает intent из свободного текста
- извлекает сумму и актив
- задает уточняющий вопрос, если запрос неполный
- формирует короткое понятное объяснение действия
- формирует advisory reply для read-only вопросов вроде:
  - `что лучше купить`
  - `объясни риски`

### Локальный код делает

- создает демо-аккаунт
- хранит баланс
- хранит историю
- хранит pending action
- считает preview по фиксированным ценам
- считает комиссию
- применяет confirm
- отклоняет невозможные операции

Это принципиально важно.

Модель не должна быть единственным источником правды для чисел.

## Preview -> Confirm -> Execute

Это центральный UX-паттерн MVP.

### Preview

Пользователь пишет:

`Купи TSLAx на 50000 RUB`

Бот:

- вызывает OpenRouter
- получает structured intent
- локально считает результат
- показывает preview

Пример preview:

`Подготовил демо-покупку TSLAx на 50 000 RUB. По демо-курсу вы получите 2.14 TSLAx. Комиссия 0.6% = 300 RUB. После операции у вас останется 450 000 RUB. Это виртуальная сделка для демонстрации UX. Подтверждаете?`

Кнопки:

- `Подтвердить`
- `Отмена`

### Confirm

При нажатии `Подтвердить`:

- бот берет `pending_action`
- еще раз проверяет баланс
- применяет сделку
- пишет запись в историю
- удаляет pending action

### Execute

Итоговое сообщение:

`Готово. Демо-операция выполнена: куплено 2.14 TSLAx за 50 000 RUB, комиссия 300 RUB. Баланс обновлен.`

## Фиксированные бизнес-правила

Чтобы не было сюрпризов в демо, все правила должны быть жестко зашиты в коде и продублированы в system prompt.

### Комиссия

- любая операция: `0.6%`

### Цены

Цены не тянем извне.

Перед демо фиксируем таблицу, например:

- `USDT = 100 RUB`
- `BTC = 8 500 000 RUB`
- `SOL = 15 000 RUB`
- `A7A5 = 1 RUB`
- `TSLAx = 23 300 RUB`
- `NVDAx = 98 000 RUB`

Можно обновлять вручную перед показом.

### Ограничения

- нельзя уйти в минус
- одна активная pending action на пользователя
- pending action живет 10 минут
- `/reset` возвращает демо-состояние к старту

## Как использовать OpenRouter

OpenRouter нужен не для вычисления портфеля, а для разговора.

Самая надежная схема:

1. бот отправляет user message + system prompt
2. модель отвечает строго JSON-объектом
3. бот валидирует JSON
4. бот локально решает, что делать дальше

### Формат ответа модели

Пример структуры:

```json
{
  "intent": "buy_asset",
  "amount": 50000,
  "payment_asset": "RUB",
  "target_asset": "TSLAx",
  "needs_clarification": false,
  "clarifying_question": null,
  "assistant_summary": "Понял запрос как демо-покупку TSLAx на 50 000 RUB."
}
```

Если запрос неполный:

```json
{
  "intent": "buy_asset",
  "amount": 50000,
  "payment_asset": null,
  "target_asset": "TSLAx",
  "needs_clarification": true,
  "clarifying_question": "В какой валюте провести покупку: RUB или USDT?",
  "assistant_summary": null
}
```

## System Prompt

В system prompt должны быть описаны:

- что бот работает только в demo-режиме
- что реальных денег и кошельков нет
- какой набор активов поддерживается
- какие интенты разрешены
- что ответ должен быть строго в JSON
- что если данных не хватает, надо задать один короткий уточняющий вопрос
- что нельзя придумывать несуществующие активы
- что нельзя обещать реальное исполнение

Но сам prompt не должен быть ledger.

Баланс и история не должны жить в prompt.

## UX-правила для жюри

У бота должна быть очень короткая и понятная коммуникация.

### Хорошо

- `Подготовил демо-обмен 10 000 RUB на USDT`
- `После операции у вас останется 490 000 RUB`
- `Это виртуальная сделка для демонстрации UX`

### Плохо

- длинные рассуждения
- финансовые дисклеймеры на пол-экрана
- технические детали про модель
- “галлюцинаторные” советы

## Схема обработки сообщения

### Если это slash-команда

Бот не ходит в модель для:

- `/start`
- `/help`
- `/balance`
- `/portfolio`
- `/history`
- `/reset`

Это должны быть прямые локальные handlers.

### Если это обычный текст

1. отправить текст в OpenRouter
2. получить JSON intent
3. если нужен clarification, отправить вопрос
4. если это `swap` или `buy_asset`, посчитать preview локально
5. записать pending action
6. показать confirm buttons

### Если это callback от кнопки

- `confirm:{pending_id}`
- `cancel:{pending_id}`

Никакой модели на этом шаге уже не нужно.

## Почему это точно проще и надежнее

Потому что в системе только два внешних зависимых контура:

- Telegram
- OpenRouter

Все остальное локально:

- состояние
- цены
- preview
- confirm
- history

Это радикально снижает шанс провала на демо.

## Railway / локальный запуск

### Самый простой вариант

Запускать локально на ноутбуке:

- `TELEGRAM_BOT_TOKEN`
- `OPENROUTER_API_KEY`
- `SQLite` файл рядом с приложением
- long polling

Это самый безопасный путь для живого demo.

### Если нужен Railway

Использовать:

- один worker service
- long polling
- volume для `SQLite`

Никаких:

- webhook
- Postgres
- Redis

для MVP не нужно.

MVP успешен, если за 2-3 минуты пользователь может:

- открыть Telegram
- получить стартовый демо-баланс
- написать финансовое намерение обычным языком
- увидеть preview
- подтвердить действие
- увидеть обновленный портфель и историю

Если это работает стабильно, то для жюри этого достаточно.

## Итоговое решение

Для этого demo не нужен operator API.

Нужен один очень тонкий Telegram-бот, в котором:

- модель через OpenRouter понимает намерение и разговаривает
- локальный код хранит демо-состояние и считает математику
- все execution делается как виртуальная операция

Это минимальная архитектура, которая выглядит как агентный финансовый продукт и при этом реально несложна в реализации.
